Render Props
¿Aún útiles?

Aunque los hooks ganaron la guerra, este patrón sigue siendo el rey para ciertos casos de reutilización visual.

Prop que es una función
Componente que recibe JSX dinámico
Definición

Una prop que devuelve JSX

<Mouse render={(mouse) => (
  <p>La posición es {mouse.x}, {mouse.y}</p>
)} />

El componente Mouse maneja el estado y llama a render con los datos. El consumidor decide qué pintar.

Seguimiento del mouse
Ejemplo

MouseTracker con render prop

function Mouse({ render }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const move = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', move);
    return () => window.removeEventListener('mousemove', move);
  }, []);
  return render(position);
}

// Uso
<Mouse render={({ x, y }) => (
  <div>Mouse: {x}, {y}</div>
)} />

El componente solo se preocupa por la lógica, la UI es 100% del consumidor.

Hooks vs Render Props
Comparación

Hooks ganaron, pero no todo

// Hook (lógica compartida)
const mouse = useMouse();
return <div>{mouse.x}</div>;

// Render prop (lógica + UI dinámica)
<Mouse>{(mouse) => <div>{mouse.x}</div>}</Mouse>

Los hooks son mejores para lógica pura. Las render props ganan cuando la estructura del JSX debe decidir el consumidor.

Casos de uso
Todavía reinan

Render Props hoy

Animaciones (React Spring, Framer Motion)

Componentes que necesitan control de renderizado (ej. <Route /> de React Router)

Reutilización de lógica + UI cuando el markup varía mucho

Composición visual sin crear componentes intermedios

Regla práctica
Conclusión

¿Render prop o hook?

¿Solo lógica? → Hook.
¿La UI varía completamente?
→ Render prop (o children como función).

No están peleados, se complementan.

Resumen: Las render props no murieron. Son ideales cuando necesitas que el consumidor controle la estructura visual mientras tú manejas el estado/efectos.
